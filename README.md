# GTK Dashboard

Веб-приложение для просмотра и анализа данных внешнеторговой деятельности Узбекистана.
Поддерживает два формата источников: Excel-файлы и DBF-файлы ГТД (грузовые таможенные декларации).

## Структура проекта

```
.
├── backend/        # FastAPI + SQLAlchemy (async) + Alembic + PostgreSQL
├── frontend/       # Next.js 16 + React 19 + Tailwind v4
├── to_import/      # DBF-данные по периодам (не коммитятся в git)
│   ├── 01.12.2020/ #   каждая подпапка = один период
│   ├── 01.2019/
│   └── …
└── GTK.md          # описание датасета
```

## Быстрый старт (локально)

### Бэкенд

Требования: Python 3.12, PostgreSQL.

```bash
cd backend
python -m venv env
source env/bin/activate          # Windows: env\Scripts\activate

pip install -r requirements.txt
cp .env.example .env             # заполнить DATABASE_URL и SECRET_KEY
```

Дальше всё через `manage.sh` (автоматически активирует venv):

```bash
./manage.sh migrate              # alembic upgrade head (первый раз и после обновлений)
./manage.sh user admin Admin123! # создать/обновить пользователя-администратора
./run.sh                         # uvicorn на :8005 (или APP_PORT из .env)
```

### Фронтенд

Требования: Node 22 (см. `frontend/.nvmrc`).

```bash
cd frontend
cp .env.example .env.local
npm install
npm run dev                      # http://localhost:3000
```

## Деплой на продакшн (Ubuntu / Debian)

### 1. Подготовка сервера

```bash
# Клонировать репозиторий
git clone <repo_url> /opt/gtk
cd /opt/gtk

# Создать venv и установить зависимости
cd backend
python3.12 -m venv env
source env/bin/activate
pip install -r requirements.txt
```

### 2. Переменные окружения

```bash
cp .env.example .env
nano .env
```

Обязательно заполнить:

```ini
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/gtk_db
SYNC_DATABASE_URL=postgresql+psycopg2://user:password@localhost:5432/gtk_db
SECRET_KEY=<сгенерировать: openssl rand -hex 32>
APP_PORT=8005
CORS_ORIGINS=https://your-domain.com
```

### 3. Первоначальная миграция и пользователь

```bash
cd /opt/gtk/backend
./manage.sh migrate
./manage.sh user admin <надёжный_пароль>
```

### 4. Запуск через systemd

```bash
sudo nano /etc/systemd/system/gtk-backend.service
```

```ini
[Unit]
Description=GTK Dashboard Backend
After=network.target postgresql.service

[Service]
User=www-data
WorkingDirectory=/opt/gtk/backend
ExecStart=/opt/gtk/backend/run.sh
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable gtk-backend
sudo systemctl start gtk-backend
```

### 5. Фронтенд (pm2 / nginx)

```bash
cd /opt/gtk/frontend
cp .env.example .env.local
# Установить NEXT_PUBLIC_API_URL=https://your-domain.com
npm install
npm run build
pm2 start npm --name gtk-frontend -- start
```

## Загрузка данных

### Формат A — Excel (.xlsx)

Стандартный формат, использовавшийся ранее. Файл содержит колонки:
`Режим`, `Дата`, `Страна`, `ТНВЕД`, `Товар`, `Категория продукции`,
`Организация Узб`, `СТИР Узб`, `Организация Хориж`, `Адрес Узб`,
`Адрес Хориж`, `Область`, `Eд.измерения`, `Вес(кг)`, `Количество`, `Цена(тыс)`.

```bash
./manage.sh load  data/gtk_data_2020.xlsx    # первая загрузка
./manage.sh reload data/gtk_data_2020.xlsx   # повторная загрузка (clean → load → enrich)
./manage.sh enrich                           # обогатить ISO-коды и регионы отдельно
```

### Формат B — DBF (ГТД / таможенные декларации)

Данные поступают в виде папок, каждая из которых соответствует одному периоду.
Структура папки (имена файлов могут отличаться, алгоритм ищет по размеру и паттерну):

```
01.12.2020/
├── База 11.dbf      ← главный файл данных (1–2 ГБ, кодировка cp1251)
├── База 11.ADX      ← индексный файл, не нужен
├── SLVS03`.DBF      ← справочник стран с ISO-кодами (кодировка cp866)  ← предпочтительный
├── SLVS03.DBF       ← справочник стран без ISO-кодов (fallback)
├── SLVS11.DBF       ← справочник единиц измерения (не обязателен)
└── 01.12.2020.APR   ← игнорируется
```

> Алгоритм определяет главный файл автоматически — по наибольшему размеру .dbf в папке.
> Справочник стран ищет по маске `SLVS03*.DBF`, предпочитает вариант со знаком `` ` ``
> (содержит ISO-2 коды). Все имена файлов могут быть любыми — главное сохранить структуру.

#### Загрузить один период

```bash
cd backend
./manage.sh dbf-load ../to_import/01.12.2020
```

Если автоопределение главного файла не сработало (нестандартное имя):

```bash
./manage.sh dbf-load ../to_import/01.12.2020 "Мои данные.dbf"
```

#### Загрузить все периоды сразу (batch-режим)

Если на сервере несколько лет данных в подпапках одной директории:

```
to_import/
├── 01.2018/
├── 01.2019/
├── 01.12.2020/
└── 01.2021/
```

```bash
cd backend
./manage.sh dbf-load-all ../to_import
```

Скрипт найдёт все подпапки, содержащие DBF-файлы, отсортирует их по имени
(хронологический порядок) и загрузит последовательно.
Если одна папка упала с ошибкой — остальные продолжат загружаться.
В конце выводится сводная таблица по всем периодам.

Пример вывода:

```
Найдено папок с ГТД: 4
  01.2018
  01.2019
  01.12.2020
  01.2021

[1/4] 01.2018
  Файл     : Base_2018.dbf  (1423 MB)
  Справочн.: SLVS03`.DBF
  OK  добавлено 987 432 / дублей 0 / некорр. 123  [8м 14с]

[2/4] 01.2019
  ...

══════════════════════════════════════════════
  Папок обработано    :      4
  Всего строк         :  4 512 000
  Добавлено           :  4 498 211
  Дублей пропущено    :     12 604
  Некорректных        :      1 185
  Общее время         : 34м 22с
══════════════════════════════════════════════
```

#### Повторный запуск безопасен

DBF-ETL использует тот же механизм дедупликации SHA-256, что и Excel-ETL.
Повторный запуск на том же файле пропустит уже существующие записи как дубли.

## Справочник команд manage.sh

| Команда | Что делает |
|---------|------------|
| `migrate` | Накатить миграции (`alembic upgrade head`) |
| `load <file.xlsx>` | Загрузить данные из Excel |
| `dbf-load <папка>` | Загрузить данные из DBF (один период) |
| `dbf-load-all <корень>` | Загрузить все периоды из подпапок |
| `clean` | Очистить таблицу `gtk` (справочники остаются) |
| `enrich` | Обогатить ISO-коды стран и регионы |
| `reload <file.xlsx>` | `clean` → `load` → `enrich` (ре-импорт Excel) |
| `stats` | Количество записей по всем таблицам |
| `user <name> <pwd>` | Создать/обновить пользователя |
| `drop` | Снести все таблицы (с подтверждением) |
| `fresh <file.xlsx>` | `drop` → `migrate` → `load` → `enrich` (с нуля) |
| `fix-prices` | **РАЗОВО** — умножить `gtk.price_thousand × 1000` (только для старых данных до патча) |

## Схема полей в базе данных

### Таблица `gtk` — основные поля (заполняются из обоих форматов)

| Поле | Тип | Описание |
|------|-----|----------|
| `regime` | enum | `ИМ` (импорт) / `ЭК` (экспорт) |
| `date` | date | Дата таможенного оформления |
| `country_id` | FK | Страна (ссылка на `countries`) |
| `product_id` | FK | Товар/ТНВЭД (ссылка на `products`) |
| `company_uzb_id` | FK | Узбекская компания |
| `company_foreign_id` | FK | Иностранная компания |
| `unit` | varchar | Единица измерения: "кг", "шт", … |
| `weight` | float | Вес нетто, кг |
| `quantity` | float | Количество в единицах |
| `price_thousand` | float | **Фактическая стоимость в USD** (не тысячах — название историческое) |
| `address_uz` | varchar | Адрес узбекской компании (только Excel) |
| `region_id` | FK | Область (только Excel, после `enrich`) |

### Дополнительные поля из DBF (Excel-записи имеют NULL)

| Поле | DBF-источник | Описание |
|------|-------------|----------|
| `declaration_number` | G7A + G7C | Номер ГТД: "26002/0008601" |
| `incoterms` | G20B | Условия поставки: CIP, FOB, EXW … |
| `incoterms_place` | G20NAME | Место поставки: "г.Ташкент" |
| `currency_code` | G22A | Валюта контракта (ISO-4217 alpha): "USD", "EUR", "RUB" |
| `currency_amount` | G22B | Сумма в валюте контракта |
| `exchange_rate` | G23 | Курс UZS / 1 ед. валюты на дату ГТД |
| `gross_weight` | G35 | Вес брутто, кг (`weight` = нетто) |
| `packages_count` | G32 | Количество мест (упаковок) |
| `customs_duty` | PAYMFACT20 | Фактически уплаченная пошлина, UZS |
| `vat_amount` | PAYMFACT29 | Фактически уплаченный НДС, UZS |

### Важно: семантика `price_thousand`

Несмотря на название, поле хранит **фактические доллары**, а не тысячи:

- **Excel**: колонка `Цена(тыс)` умножается на 1000 при загрузке → `price_thousand`
- **DBF**: поле `G45USD` (CIF-стоимость в USD) записывается напрямую → `price_thousand`
  - `G45USD` — статистическая (CIF) стоимость: включает страховку и фрахт, всегда ≥ контрактной цены
  - Если `G45USD = 0`, fallback: `G22B` (контрактная цена, только когда `G22A = "840"` = USD)

## Возможности интерфейса

- `/dashboard/gtk` — постраничная таблица записей с фильтрами (страна, ТНВЭД, режим, даты, поиск).
- `/dashboard/charts` — карта мира, помесячные графики, топ-чарты, итоговые карточки. Поддерживает wildcard `707*`.
- `/dashboard/users` — управление пользователями (только для администраторов).
- `/dashboard/upload` — загрузка Excel-файла из браузера (только администратор). Дубликаты отсекаются автоматически.
- `POST /api/admin/upload-gtk-dbf` — загрузка DBF-файла или ZIP-архива через API (до 2 ГБ).

## Частые вопросы

**Q: Папки на сервере называются иначе, чем `01.12.2020` — это проблема?**
A: Нет. `dbf-load-all` определяет главный файл по размеру внутри папки, а не по имени папки или файла.

**Q: В папке нет SLVS03.DBF — что будет?**
A: Коды стран из поля G15 (например, "156") будут записаны как названия стран.
Данные загрузятся, но названия стран будут числовыми кодами.
Рекомендуется всегда держать SLVS03.DBF рядом с главным файлом.

**Q: Можно ли смешивать Excel- и DBF-данные в одной БД?**
A: Да. Оба формата используют одну таблицу `gtk` с одинаковым механизмом дедупликации.
Excel-записи будут иметь NULL в DBF-специфичных полях, DBF-записи — NULL в `address_uz`/`region_id`.

**Q: Как убедиться что данные загружены корректно?**
A: Запустите `./manage.sh stats` — покажет счётчики по всем таблицам.
Также проверьте через Swagger UI на `http://localhost:8005/docs`.
